## 1. Назначение и общее описание

Backuper — система резервного копирования одной директории клиента на сервер с корзиной, проверкой целостности по размеру, алертами на почту и работой в роли системной службы.

- **Клиент** хранит рабочие данные в одной директории (например `/data`) со вложенными поддиректориями любой глубины. Объём ~4 ТБ, ~1 млн файлов (в основном отсканированные PDF, иногда крупные видео).
- **Сервер** периодически инициирует сверку, скачивает с клиента новые и изменённые файлы, удалённые на клиенте файлы переносит в корзину, по истечении срока корзину очищает.
- Связь — собственный бинарный протокол поверх **сырых TLS-сокетов** (не WebSocket), с взаимной аутентификацией (mTLS) и обменом паролями.
- Обо всех значимых событиях и **любых ошибках** уведомляет по e-mail (SMTP).

**Направление:** инициатор — **сервер** (модель pull). Сервер подключается к клиенту, запрашивает список файлов и скачивает их содержимое. Клиент — служба, слушающая TLS-порт.

**Основной поток:** клиент → сервер. Дополнительно поддерживается **восстановление** (restore) сервер → клиент (раздел 13).

## 3. Технологический стек 🔧

**Язык: Go.**
Обоснование: компилируемый и быстрый (важно при 1 млн файлов / 4 ТБ), но проще Rust; горутины для параллельной передачи; единый статический бинарник без рантайм-зависимостей; кросс-компиляция под Debian/Linux и Windows; зрелый `crypto/tls` (1.2/1.3); готовые библиотеки для systemd и Windows Service.

- **ОС:** сервер и клиент — Debian-семейство Linux **или** Windows 10+. Код кроссплатформенный (пути через `path/filepath`, относительные relpath нормализуются к `/`).
- **Хранение состояния/индекса:** SQLite (режим WAL) — компактно, транзакционно, переживает перезапуск, тянет индекс ~1 млн записей и очередь задач.
- **Конфигурация:** `.env` (отдельные файлы для сервера и клиента).
- **Зависимости 🔧:** минимальный набор — драйвер SQLite, библиотека ротации логов, отправка SMTP (стандартная `net/smtp` либо лёгкая обёртка для HTML/STARTTLS). Список фиксируется на старте реализации.

## 4. Архитектура и модель параллелизма

### 4.1 Общая схема

```
            (сервер инициирует подключение, одна LAN)
   ┌──────────────────┐   TLS 1.2+, mTLS, свой протокол   ┌──────────────────┐
   │      СЕРВЕР       │ ─────────  control conn  ───────► │      КЛИЕНТ       │
   │   (служба)        │  AUTH/LIST/DISK/PING               │   (служба)        │
   │                   │ ◄──────────────────────────────── │                   │
   │  storage_dir      │   N× data conn (пул)              │   backup_dir      │
   │  trash_dir        │ ─────────  GET(offset)  ─────────► │    (/data)        │
   │  temp_dir         │ ◄────────  file data    ────────── │                   │
   │  SQLite (WAL)     │   PUT (restore)                    │                   │
   │  SMTP alerts      │                                    │                   │
   └──────────────────┘                                    └──────────────────┘
```

### 4.2 Модель параллелизма 🔧
- **Контрольное соединение** — одно TLS-соединение, живущее на время цикла: служебные команды `LIST`, `DISK`, периодический `PING`.
- **Пул соединений передачи** — до `PARALLEL_TRANSFERS` независимых TLS-соединений. Каждое в любой момент качает **один** файл целиком (или докачивает с offset). Параллелизм достигается несколькими соединениями, а не мультиплексированием одного — это проще и надёжнее.
- Каждое соединение проходит собственное рукопожатие (TLS + обмен паролями).
- Глобальный лимит скорости (`BANDWIDTH_LIMIT`) делится между всеми соединениями (общий token-bucket).

### 4.3 Компоненты

**Сервер:**
- `scheduler` — запуск циклов по интервалу, постановка в очередь при наложении.
- `differ` — сравнение списка клиента с индексом хранилища.
- `transfer manager` — пул соединений, докачка, проверка размера, лимит скорости.
- `trash manager` — перенос в корзину, версии, очистка по сроку.
- `alert manager` — агрегация событий и отправка HTML-писем.
- `store` — SQLite: индекс, очередь, корзина, статистика, флаги.
- `lock` — защита одиночного запуска.
- `logx` — логирование + аудит.
- `diskinfo` — свободное место сервера.

**Клиент:**
- `tls listener` — приём соединений, рукопожатие, маршрутизация команд.
- `scanner` — обход источника с фильтрами, выдача списка батчами.
- `file server` — отдача содержимого с offset; приём (restore).
- `diskinfo` — свободное место диска источника.
- `logx` — логирование + аудит.

## 5. Протокол обмена (полная спецификация)

### 5.1 Транспорт
- TCP → TLS (минимум 1.2) → mTLS → обмен паролями → команды.
- Все числа — **big-endian**. Строки — `[uint32 длина][UTF-8 байты]`. Время (mtime) — `int64` Unix-наносекунды.

### 5.2 Формат кадра
```
смещение  размер  поле
0         1       версия протокола (uint8)         = 1
1         1       тип сообщения (uint8)            см. 5.4
2         8       длина полезной нагрузки (uint64)
10        N       полезная нагрузка
```
Максимальный размер одного кадра данных (чанка) — `CHUNK_SIZE` (по умолчанию 1 МиБ, env). Метаданные (LIST батчи) ограничены `MAX_FRAME_BYTES` 🔧 (по умолчанию 16 МиБ) для защиты от переполнения.

### 5.3 Последовательность рукопожатия
```
1. TCP connect (server → client)
2. TLS handshake: обе стороны предъявляют сертификат, проверяют по общему CA (mTLS)
3. server → client : AUTH_REQ { proto_version, own_password }
4. client проверяет пароль == PEER_PASSWORD
   client → server : AUTH_RESP { own_password, status }
5. server проверяет пароль клиента == PEER_PASSWORD
6. при несовпадении любой стороны → ERROR{code=AUTH_FAILED} + разрыв + алерт
7. сессия установлена; далее команды
```

### 5.4 Типы сообщений и формат payload

| Код | Имя | Напр. | Payload |
|---|---|---|---|
| 0x01 | AUTH_REQ | S→C | `proto_version:uint8`, `password:string` |
| 0x02 | AUTH_RESP | C→S | `password:string`, `status:uint8` (0=ok) |
| 0x10 | LIST_REQ | S→C | `root:string` (пусто = весь источник) |
| 0x11 | LIST_BATCH | C→S | `is_last:uint8`, `count:uint32`, затем `count`× `{path:string, size:uint64, mtime:int64}` |
| 0x20 | GET_REQ | S→C | `path:string`, `offset:uint64` |
| 0x21 | GET_RESP | C→S | `status:uint8`, `total_size:uint64`, `mtime:int64` (метаданные перед данными) |
| 0x22 | FILE_DATA | C→S / S→C | `bytes` (чанк, ≤ CHUNK_SIZE) |
| 0x23 | FILE_END | C→S / S→C | пусто (сигнал конца потока файла) |
| 0x30 | PUT_REQ | S→C | `path:string`, `total_size:uint64`, `offset:uint64`, `mtime:int64` |
| 0x31 | PUT_RESP | C→S | `status:uint8`, `resume_offset:uint64` (с какого байта клиент готов принять) |
| 0x40 | DISK_REQ | S→C | пусто |
| 0x41 | DISK_RESP | C→S | `count:uint8`, затем `count`× `{mount:string, total:uint64, free:uint64}` |
| 0xF0 | PING | S↔C | пусто |
| 0xF1 | PONG | S↔C | пусто |
| 0xFF | ERROR | любое | `code:uint16`, `message:string` |

Поток скачивания файла: `GET_REQ` → `GET_RESP` → серия `FILE_DATA` → `FILE_END`. Получатель знает `total_size` и контролирует завершение. Restore — зеркально через `PUT_*`.

### 5.5 Коды ошибок (ERROR.code)

| Код | Имя | Значение |
|---|---|---|
| 1 | AUTH_FAILED | неверный пароль / mTLS-несоответствие |
| 2 | NOT_FOUND | файл/путь не найден |
| 3 | PERMISSION_DENIED | нет прав на чтение/запись |
| 4 | IO_ERROR | ошибка ввода-вывода |
| 5 | DISK_FULL | недостаточно места |
| 6 | BAD_OFFSET | offset больше размера файла / некорректен |
| 7 | BUSY | файл занят/заблокирован |
| 8 | PROTOCOL_ERROR | нарушение формата/версии |
| 9 | SIZE_MISMATCH | размер после передачи не совпал |
| 10 | UNSUPPORTED | неизвестный тип сообщения/команда |

Любая ERROR логируется и попадает в агрегированный алерт.

### 5.6 Идентификация файла и пути
- Файл идентифицируется **relpath** от корня (`/`-разделитель, без ведущего слэша).
- Структура поддиректорий копируется в хранилище **1 в 1**.
- **Минимальная чувствительность к регистру** 🔧: при сравнении relpath нормализуется к нижнему регистру (для связки Windows↔Linux); при сохранении используется оригинальное имя. Если у клиента есть два пути, различающиеся только регистром, — это коллизия: второй пропускается с алертом (крайне маловероятно, но фиксируем).
- **Запрет выхода за корень:** клиент проверяет, что запрошенный path не выходит за `backup_dir` (защита от `..`); сервер так же при restore/сохранении.

### 5.7 Таймауты протокола
- `CONNECT_TIMEOUT` — на установку TCP+TLS+auth (по умолчанию 30s).
- `IO_TIMEOUT` — дедлайн на чтение/запись каждого кадра (по умолчанию 60s); сбрасывается при прогрессе.
- `PING` каждые `HEALTHCHECK_INTERVAL` (по умолчанию 30s) по контрольному соединению; нет `PONG` за `IO_TIMEOUT` → соединение считается разорванным.

## 6. Безопасность

### 6.1 TLS / mTLS
- Минимум **TLS 1.2** (`TLS_MIN_VERSION`), допускается 1.3.
- **Самоподписанные** сертификаты, **взаимная аутентификация**: и сервер, и клиент проверяют сертификат другой стороны по общему CA (`TLS_CA_FILE`).
- 🔧 Параметры генерации сертификатов: RSA-2048 или ECDSA P-256; срок ~3650 дней; в SAN сервера/клиента — их IP/hostname в LAN. Генерация — в установочном скрипте (раздел 24).

### 6.2 Обмен паролями (защита от подмены)
- У сервера: `OWN_PASSWORD` (отправляет в AUTH_REQ) и `PEER_PASSWORD` (ожидает от клиента).
- У клиента — зеркально (`OWN_PASSWORD` = его пароль, `PEER_PASSWORD` = пароль сервера).
- Соответствие: `server.OWN_PASSWORD == client.PEER_PASSWORD` и `client.OWN_PASSWORD == server.PEER_PASSWORD`.
- Пароли — в `.env` в открытом виде (по согласованию). Рекомендуемая длина ≥ 24 символа.
- Несовпадение → ERROR{AUTH_FAILED}, разрыв, лог, **алерт**.

### 6.3 Прочее
- Replay-защита и шифрование «в покое» (at rest) — **не требуются** (достаточно шифрования канала).
- Файлы `.env` и ключи — права `600`, владелец — служебный пользователь.
- Ротация ключей/паролей — вручную: заменить файлы/значения и перезапустить службы.

## 7. Логика сверки и синхронизации

### 7.1 Цикл сверки (псевдокод)
```
on schedule tick:
    if cycle already running: enqueue one; send alert; return
    acquire lock
    cycle = new cycle record (started_at = now MSK)
    try:
        conn = connect_control()                 # TLS + auth, retry x RETRY_COUNT
        disk_client = conn.DISK()                 # свободное место клиента
        check_disk_thresholds(server, client)     # >90% → немедленный алерт
        for pass in 1..SYNC_MAX_PASSES:
            client_list = stream_list(conn)        # LIST батчами → temp table
            diff = compute_diff(client_list, index)   # см. раздел 8
            enqueue(diff.to_download)               # tasks
            move_to_trash(diff.to_trash)            # см. раздел 11
            run_transfer_queue()                   # пул соединений, докачка, retry
            if diff.to_download == 0 and diff.to_trash == 0:
                break                              # сошлось раньше
        cleanup_trash_if_due()                     # по своему интервалу
        cycle.status = OK (или PARTIAL если остались расхождения)
    catch err:
        cycle.status = FAILED; record error
    finally:
        release lock
        finalize cycle stats
        send_report_email(cycle)                   # каждый цикл
```

### 7.2 Критерий «файл изменился»
Сравнение по **relpath + размер (байты) + mtime**:
- **новый**: есть у клиента, нет в индексе → скачать;
- **изменённый**: relpath совпал, но `size` отличается **или** `|mtime_client − mtime_index| > MTIME_TOLERANCE` → скачать заново;
- **без изменений**: совпали size и mtime (в пределах допуска) → пропустить.
- Хэши **не считаются**.

**MTIME_TOLERANCE** 🔧 (по умолчанию 2s): сглаживает разную точность mtime у файловых систем (NTFS/ext4/FAT) и часовые расхождения. **После скачивания сервер выставляет mtime сохранённой копии равным mtime клиента** и пишет это значение в индекс — иначе следующий цикл считал бы файл изменённым.

### 7.3 Переименование
Отдельное детектирование не делается: старое имя пропадает у клиента → его копия уходит в корзину; новое имя появляется → скачивается в активную директорию.

### 7.4 Особые случаи
- **Вложенные директории** копируются 1 в 1 на любую глубину.
- **Пустые директории не копируются**; директория в хранилище создаётся только когда внутри появляется файл.
- **Symlink-и игнорируются** (не следуем, не копируем; в логе — debug-запись).
- **Заблокированный/занятый файл**: чтение/сохранение пробуется `RETRY_COUNT` раз с паузой `RETRY_DELAY`; при неудаче — **алерт**, файл помечается `skipped`, цикл продолжается.
- **Часто меняющийся файл**: та же логика 3 попыток; если за `SYNC_MAX_PASSES` так и не стабилизировался — остаётся на следующий цикл.

### 7.5 Фильтры
- Списки **включения/исключения** по glob-маскам в `.env` клиента (применяются сканером при построении списка). По умолчанию фильтров нет. Пример исключения: `*.tmp, *.lock, ~$*`.

### 7.6 Права на сохраняемые файлы 🔧
- При сохранении на сервере: файлы — `SAVE_FILE_PERMS` (по умолчанию `0666`), директории — `SAVE_DIR_PERMS` (по умолчанию `0777`). Можно ужесточить.

## 8. Алгоритм сравнения (дифф)

Чтобы не держать 1 млн записей в памяти, список клиента стримится в **временную таблицу** SQLite, затем дифф — SQL-операциями над множествами.

```
-- приём списка
BEGIN;
DELETE FROM client_files;                 -- очистить временную таблицу
-- для каждого LIST_BATCH:
INSERT INTO client_files(relpath, relpath_norm, size, mtime) VALUES (...);  -- батчем
COMMIT;

-- новые и изменённые (к скачиванию)
SELECT c.relpath, c.size, c.mtime
FROM client_files c
LEFT JOIN files f ON f.relpath_norm = c.relpath_norm
WHERE f.relpath_norm IS NULL
   OR f.size <> c.size
   OR ABS(f.mtime - c.mtime) > :tol;

-- удалённые у клиента (в корзину)
SELECT f.relpath, f.size
FROM files f
LEFT JOIN client_files c ON c.relpath_norm = f.relpath_norm
WHERE c.relpath_norm IS NULL;
```

- `client_files` индексируется по `relpath_norm`; `files` тоже. Сравнение — по нормализованному (нижний регистр) пути.
- Результаты диффа материализуются в очередь `tasks` (скачивания) и в операции корзины.
- Подсчёт количества для порога массового удаления — `COUNT(*)` второго запроса.

## 9. Передача файлов

### 9.1 Параллельность и лимит
- До `PARALLEL_TRANSFERS` соединений качают одновременно (пул).
- `BANDWIDTH_LIMIT` (байт/с, общий) — token-bucket, делится между соединениями; `0` = без лимита. Лимит можно задавать с обеих сторон; эффективен меньший.
- Сжатие не используется.

### 9.2 Атомарность
- Файл качается во временный файл в `TEMP_DIR`: имя 🔧 `sha1(relpath).part` (стабильно между перезапусками для докачки) + рядом `*.meta` (relpath, total_size, mtime).
- После завершения и совпадения размера: `fsync` → атомарный `rename` в целевой путь в `STORAGE_DIR` → установка mtime = mtime клиента → запись/обновление в индексе.
- Недостающие директории создаются заранее.

### 9.3 Докачка
- При обрыве частичный `*.part` сохраняется; в `tasks` хранится `offset = размер .part`.
- Возобновление: `GET_REQ{path, offset}` → клиент шлёт `GET_RESP{total_size, mtime}` и поток с `offset`.
- Если на клиенте файл за это время изменился (отличается размер/mtime) → `*.part` отбрасывается, скачивание начинается заново с 0.
- При старте службы сканируются `tasks` со статусом `pending/in_progress` → возобновляются.

### 9.4 Состояние задачи (state machine)
```
pending ──► in_progress ──► done
                │   │
                │   └──► failed ──(attempts<RETRY_COUNT)──► pending (через RETRY_DELAY)
                │                └─(attempts==RETRY_COUNT)─► skipped (+ алерт)
                └──► (обрыв) сохраняет offset, остаётся in_progress/pending
```

## 10. Проверка целостности

- После переноса проверяется **совпадение размера в байтах** между источником (`total_size` из `GET_RESP`) и сохранённой копией.
- Хэш **не используется** (нагрузка недопустима на 4 ТБ).
- Несовпадение размера (`SIZE_MISMATCH`) → повтор скачивания до `RETRY_COUNT` раз; все неуспешны → **алерт**, файл `skipped`, цикл продолжается.
- Фоновый scrubbing не требуется.

## 11. Корзина

### 11.1 Перенос
- Файл, удалённый на клиенте, **перемещается** (внутри сервера: storage → trash; по возможности `rename`, иначе копирование+удаление) в `TRASH_DIR` с **сохранением оригинального relpath**.
- Целевой путь: `TRASH_DIR/<relpath>`.
- В таблице `trash` фиксируется `relpath`, `version`, `trashed_at` (МСК), `size`.

### 11.2 Несколько версий
- Если на момент переноса `TRASH_DIR/<relpath>` уже занят прежней версией — прежняя перемещается в скрытую подпапку версий `TRASH_DIR/<dir>/.versions/<seq>/<filename>`, а новая занимает канонический путь. **Имя файла не искажается** (метка не добавляется к имени — она только в имени служебной подпапки). Каждая версия имеет свой `trashed_at` и истекает независимо.

### 11.3 Срок и очистка
- `TRASH_RETENTION_DAYS` (по умолчанию 10), отсчёт от `trashed_at`. Если файл за это время вручную не извлекли — удаляется безвозвратно. Иные «касания» не отслеживаются, только таймер.
- Очистка запускается по `TRASH_CLEANUP_INTERVAL` (env): `DELETE` файлов с `trashed_at < now − retention`, удаление с диска, запись в аудит, учёт в статистике письма.
- Лимит размера корзины — не нужен. Алерт перед удалением — не нужен.

### 11.4 Повторное появление
- Если удалённый (уже в корзине) файл снова появляется на клиенте — он **скачивается заново** в активную директорию (не восстанавливается из корзины). Версия в корзине доживает свой срок.

### 11.5 Защита от массового удаления
- Если за цикл в корзину уходит больше `ALERT_MASS_DELETE_THRESHOLD` файлов (по умолчанию 100) — **алерт** (перенос всё равно выполняется).

## 12. Первый запуск

- «Первый запуск» определяется по `meta.initialized != 1` (и/или пустому индексу `files`).
- Полный список специально не запрашивается — он берётся обычным `LIST` из `backup_dir`. Сервер:
  1. **Индексирует** то, что перенесено вручную в `STORAGE_DIR` (обход, заполнение `files`: relpath, size, mtime). Symlink-и игнорируются.
  2. Запрашивает `LIST` у клиента.
  3. Сверяет по **relpath / size / mtime** (без хэшей), с тем же `MTIME_TOLERANCE`.
  4. Появившиеся/изменившиеся за время переноса файлы — докачивает.
  5. Файлы, что есть в хранилище, но отсутствуют у клиента, — **переносит в корзину**.
  6. Ставит `meta.initialized = 1`.
- Так как ручной перенос мог идти долго и менять mtime — индексирование хранилища берёт фактические size/mtime файлов на сервере.

---

## 13. Восстановление (restore)

- Направление: сервер → клиент. Цель — тот же `backup_dir` (источник и место восстановления совпадают).
- Запуск **вручную** командой на сервере (раздел 22): весь набор, поддерево или один файл.
- Передача через `PUT_REQ/PUT_RESP/FILE_DATA/FILE_END`:
  - клиент в `PUT_RESP` сообщает `resume_offset` (для докачки restore);
  - запись во временный файл на клиенте → проверка размера → атомарный rename → установка mtime;
  - те же гарантии: retry ×N, проверка размера, алерты.
- 🔧 По умолчанию restore **не перезаписывает** более новый файл на клиенте без флага `--force` (защита от затирания свежих данных). Поведение настраивается флагом команды.

## 14. Повторные попытки и тайм-ауты

- `RETRY_COUNT` (по умолчанию 3), `RETRY_DELAY` (по умолчанию 1m), интервал фиксированный.
- Применяется ко всем операциям: соединение, LIST, GET, сохранение, PUT (restore).
- По файлу: после `RETRY_COUNT` неудач — `skipped` + алерт, цикл продолжается.
- На уровне цикла/сети (клиент недоступен, обрыв): `RETRY_COUNT` попыток, затем алерт и продолжение **со следующего планового цикла** (не раньше).
- После неуспешного цикла — следующий запуск строго по плану.
- Тайм-ауты: `CONNECT_TIMEOUT`, `IO_TIMEOUT`, `HEALTHCHECK_INTERVAL` (раздел 5.7).

## 15. Алерты по e-mail

### 15.1 Транспорт
- Все параметры в `.env`: `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASSWORD`, `SMTP_FROM`, `SMTP_TO` (несколько адресов через запятую), `SMTP_SECURITY` (`none`/`starttls`/`tls`).
- Формат письма — **HTML** (таблица со статистикой) + текстовая версия (fallback).
- При сбое самой отправки письма — запись в лог уровня ERROR (письмом сообщить нельзя).

### 15.2 Триггеры

| Событие | Когда | Срочность |
|---|---|---|
| Отчёт по циклу | после каждого цикла | планово |
| Любая ошибка | права, IO, диск, сеть, протокол, размер | в составе отчёта/агрегата |
| Провал 3 попыток | по операции/файлу | в составе агрегата |
| Массовое удаление | > порога за цикл | планово (с отчётом) |
| Диск > 90% | сервер или клиент | **немедленно, вне расписания** |
| Перезапуск службы | при старте после падения | немедленно |
| Очередь наложилась | новый цикл при активном | немедленно |

### 15.3 Агрегация
- Множество событий одного цикла/окна объединяются в **одно письмо** (антиспам). 🔧 Немедленные алерты (диск, перезапуск) шлются отдельно, но при всплеске — группируются в пределах окна `ALERT_AGGREGATION_WINDOW` (по умолчанию 60s).

### 15.4 Содержимое отчёта (максимально полно)
- Период: `started_at` / `finished_at` / длительность (МСК).
- Статус цикла: OK / PARTIAL / FAILED.
- Скачано файлов и **байт**; изменено файлов.
- Перенесено в корзину; удалено из корзины по сроку.
- Число проходов (passes), и с какого прохода/попытки сошлись проблемные файлы.
- Пропущено файлов (`skipped`) с причинами.
- **Свободно (%)** на дисках **клиента и сервера** (и абсолютные ГБ).
- Скорость передачи (средняя), пиковая параллельность.
- Список ошибок цикла (код, путь, сообщение).

### 15.5 Шаблон письма (структура) 🔧
```
Тема: [Backuper] Цикл #<id> — <STATUS> — скачано <N> файлов (<GB>)
─────────────────────────────────────────────
Сводка:        статус, период, длительность
Передача:      скачано / изменено / байт / скорость / проходы
Корзина:       перенесено / удалено по сроку
Диски:         сервер X% свободно (A/B ГБ), клиент Y% свободно (C/D ГБ)
Ошибки:        таблица (код | путь | сообщение)
Пропущено:     таблица (путь | причина | попыток)
```

## 16. Логирование и аудит

- Логи ведутся **и на сервере, и на клиенте**.
- `LOG_LEVEL` (DEBUG/INFO/WARN/ERROR) и параметры логов — в `.env`.
- Назначение и ротация: `LOG_DIR`, `LOG_MAX_SIZE`, `LOG_MAX_FILES`, `LOG_MAX_AGE` 🔧. По умолчанию — файлы с ротацией по размеру.
- **Аудит-лог** (`AUDIT_LOG`) операций с файлами в формате **JSON** (по строке на событие):
  ```json
  {"ts":"2026-06-24T14:05:00+03:00","actor":"server","op":"download",
   "relpath":"docs/scan_001.pdf","size":1048576,"result":"ok","attempt":1,"cycle":42}
  ```
  Операции аудита: `download`, `change`, `trash`, `trash_purge`, `restore`, `skip`.
- Основной лог 🔧 — человекочитаемый текст (`время уровень компонент сообщение`), переключаемый на JSON через env.
- Пример строки:
  ```
  2026-06-24 14:05:00 INFO  transfer  downloaded docs/scan_001.pdf (1.0 MiB) in 0.4s [conn=2]
  ```

## 17. Хранение состояния (схема SQLite)

База на сервере, режим **WAL**, единый файл `STATE_DB`.

```sql
-- индекс файлов хранилища
CREATE TABLE files (
  relpath       TEXT PRIMARY KEY,   -- оригинальный путь (/)
  relpath_norm  TEXT NOT NULL,      -- нижний регистр для сравнения
  size          INTEGER NOT NULL,
  mtime         INTEGER NOT NULL,   -- Unix ns (МСК-семантика хранится как UTC ns)
  updated_at    INTEGER NOT NULL
);
CREATE INDEX idx_files_norm ON files(relpath_norm);

-- временная таблица списка клиента (очищается каждый проход)
CREATE TABLE client_files (
  relpath       TEXT,
  relpath_norm  TEXT,
  size          INTEGER,
  mtime         INTEGER
);
CREATE INDEX idx_client_norm ON client_files(relpath_norm);

-- очередь передачи (персистентная)
CREATE TABLE tasks (
  id          INTEGER PRIMARY KEY,
  relpath     TEXT NOT NULL,
  kind        TEXT NOT NULL,        -- download | restore
  size        INTEGER NOT NULL,
  offset      INTEGER NOT NULL DEFAULT 0,
  status      TEXT NOT NULL,        -- pending|in_progress|done|failed|skipped
  attempts    INTEGER NOT NULL DEFAULT 0,
  last_error  TEXT,
  cycle_id    INTEGER,
  created_at  INTEGER NOT NULL,
  updated_at  INTEGER NOT NULL
);
CREATE INDEX idx_tasks_status ON tasks(status);

-- корзина
CREATE TABLE trash (
  id          INTEGER PRIMARY KEY,
  relpath     TEXT NOT NULL,
  version     INTEGER NOT NULL DEFAULT 0,  -- 0 = канонический путь
  size        INTEGER NOT NULL,
  trashed_at  INTEGER NOT NULL
);
CREATE INDEX idx_trash_time ON trash(trashed_at);

-- циклы и статистика
CREATE TABLE cycles (
  id              INTEGER PRIMARY KEY,
  started_at      INTEGER NOT NULL,
  finished_at     INTEGER,
  status          TEXT,             -- OK|PARTIAL|FAILED
  passes_used     INTEGER,
  downloaded_files INTEGER DEFAULT 0,
  downloaded_bytes INTEGER DEFAULT 0,
  changed_files   INTEGER DEFAULT 0,
  trashed_files   INTEGER DEFAULT 0,
  purged_files    INTEGER DEFAULT 0,
  skipped_files   INTEGER DEFAULT 0,
  error_summary   TEXT
);

-- события/алерты для агрегации
CREATE TABLE events (
  id          INTEGER PRIMARY KEY,
  cycle_id    INTEGER,
  severity    TEXT,                 -- INFO|WARN|ERROR
  type        TEXT,                 -- auth|io|disk|size|mass_delete|restart|overlap
  relpath     TEXT,
  message     TEXT,
  created_at  INTEGER NOT NULL,
  sent        INTEGER NOT NULL DEFAULT 0
);

-- метаданные/флаги
CREATE TABLE meta ( key TEXT PRIMARY KEY, value TEXT );
-- meta: initialized=1, schema_version=N, last_cleanup_at=...
```

- Блокировка одиночного запуска — отдельный `LOCK_FILE` (advisory lock / pid-файл).
- Состояние клиента простое (stateless по бэкапу), отдельная БД не нужна; клиент держит только логи/аудит.

## 19. Служба (systemd / Windows Service)

### 19.1 Linux (systemd)
- Юниты `backuper-server.service` и `backuper-client.service`.
- 🔧 Запуск под служебным пользователем `backuper` (не root), с доступом к нужным директориям.
- Ключевые директивы: `Restart=always`, `RestartSec=5`, `User=backuper`, `WorkingDirectory`, `EnvironmentFile=/etc/backuper/.env`, `ExecStart=…`, `TimeoutStopSec` для graceful.
- **Алерт о перезапуске**: при старте служба определяет, что предыдущий выход был аварийным (по флагу чистого завершения в `meta`/файле состояния), и шлёт письмо.

### 19.2 Windows 10+
- Регистрация как Windows Service (через менеджер сервисов). Автозапуск, автоперезапуск при сбое (recovery actions). `.env` рядом с бинарником.

### 19.3 Общее
- **Healthcheck**: команда статуса (`backuper-* status` / `systemctl status`) + PING на контрольном соединении; время последнего успешного цикла пишется в `meta`/статус-файл.
- **Graceful shutdown**: по сигналу остановки текущая передача либо завершается, либо фиксирует offset; очередь и состояние сохраняются; lock освобождается.

## 20. Производительность и масштабирование (1 млн файлов / 4 ТБ)

- **Список** передаётся и обрабатывается **батчами** (`LIST_BATCH`, напр. 10 000 записей в кадре), пишется в `client_files` пакетными `INSERT` в одной транзакции. Память не растёт линейно от числа файлов.
- **Дифф** — SQL-джоинами по индексу `relpath_norm`, без загрузки всего в память.
- **SQLite**: WAL, `PRAGMA synchronous=NORMAL`, батч-транзакции, индексы по путям.
- **Обход директорий** — потоковый (`filepath.WalkDir`), без накопления полного среза в RAM.
- **Передача**: `PARALLEL_TRANSFERS` соединений, чанки `CHUNK_SIZE`, `io.Copy` с буфером; крупные видео качаются по чанкам с докачкой.
- **Первый запуск** (индексирование 4 ТБ): только `stat` (size/mtime), без чтения содержимого и без хэшей — быстро.
- 🔧 Оценка: при гигабитной LAN (~100–110 МБ/с) полная передача 4 ТБ ≈ 10–11 часов в один поток; параллелизм и отсутствие сжатия помогают приблизиться к пределу сети. Это ориентир для интервала первого досверочного прогона.

## 21. Граничные случаи (матрица)

| Ситуация | Поведение |
|---|---|
| Нет места на сервере при скачивании | `DISK_FULL` → алерт + пауза цикла |
| Диск (клиент/сервер) > порога | немедленный алерт вне расписания |
| Потеря сети в цикле | retry ×N → алерт → продолжить со след. планового цикла |
| Клиент недоступен в плановое время | retry ×N → алерт → продолжить со след. цикла |
| Два экземпляра службы | запрещено через lock-файл |
| Новый цикл при активном | поставить в очередь (1 шт.) + алерт |
| Массовое удаление (> порога) | в корзину + алерт |
| Заблокированный/занятый файл | retry ×N → пропуск + алерт |
| Файл изменился во время докачки | сброс `.part`, скачать заново с 0 |
| Коллизия путей по регистру | второй путь пропустить + алерт |
| Symlink | игнор (debug-лог) |
| Сбой отправки письма | ERROR в лог |
| Часовые расхождения / точность mtime | сглаживается `MTIME_TOLERANCE`; mtime копии = mtime клиента |

## 22. CLI-команды 🔧

**Сервер:**
- `backuper-server run` — основной режим (служба).
- `backuper-server check-config` — валидация `.env` и доступности ресурсов.
- `backuper-server dry-run` — цикл сверки без передачи: показать, что было бы скачано/в корзину.
- `backuper-server restore --path <relpath|dir> [--all] [--force]` — восстановление на клиент.
- `backuper-server status` — последний цикл, состояние очереди, healthcheck.
- `backuper-server test` — тестовый прогон всех функций (раздел 23).
- `backuper-server gen-certs` — генерация самоподписанных CA/сертификатов (или отдельный скрипт).

**Клиент:**
- `backuper-client run` — служба (listener).
- `backuper-client check-config` — валидация `.env`.
- `backuper-client status` — состояние слушателя, последние соединения.

## 23. Тестирование и dry-run

### 23.1 Dry-run
Прогон цикла без реальной передачи: вывод списков «к скачиванию» и «в корзину», проверка соединения, mTLS и паролей, доступности SMTP (тестовое письмо опционально).

### 23.2 Тестовый сценарий «все функции» (`scripts/test-all`) 🔧
Поднять временные `backup_dir`/`storage_dir`/`trash_dir`, сгенерировать тестовые сертификаты, запустить обе стороны локально и проверить:
1. Рукопожатие TLS + mTLS + пароли (и негатив: неверный пароль → AUTH_FAILED + алерт).
2. Первый запуск: сверка ручного переноса (часть файлов совпадает, часть новая/изменённая, часть лишняя → корзина).
3. Скачивание нового и изменённого файла.
4. Сравнение по size/mtime + допуск; неизменный файл не качается повторно.
5. Докачка после принудительного обрыва (offset > 0).
6. Изменение файла во время докачки → перекачка с 0.
7. Проверка размера: подмена размера → `SIZE_MISMATCH` → retry → skip + алерт.
8. Удаление на клиенте → перенос в корзину (с сохранением пути) + версии при повторном удалении.
9. Очистка корзины по сроку (с подменой `trashed_at`).
10. Массовое удаление > порога → алерт.
11. Диск > порога → немедленный алерт.
12. Retry/таймауты: недоступный клиент → 3 попытки → алерт → продолжение.
13. Restore одного файла и поддерева (+ поведение `--force`).
14. Наложение циклов → очередь + алерт.
15. Перезапуск службы с недокачанной очередью → возобновление.
16. Формирование и (через mock-SMTP) отправка HTML-отчёта со всей статистикой.

## 24. Деплой (пошагово)

Формат поставки: **исходники + установочный скрипт**. Инструкция — `DEPLOY.md` (на русском), отдельно для сервера и клиента, Linux и Windows.

### 24.1 Сборка
```
# на машине сборки (Go)
make build            # или: go build ./cmd/server && go build ./cmd/client
# кросс-компиляция под Windows при необходимости:
GOOS=windows GOARCH=amd64 go build ./cmd/client
```

### 24.2 Сертификаты (один раз)
```
deploy/gen-certs.sh   # создаёт CA, server.crt/key, client.crt/key с SAN = IP в LAN
# CA-файл (ca.crt) кладётся обеим сторонам; приватные ключи — только своей стороне
```

### 24.3 Сервер (Linux)
```
sudo deploy/install.sh server
# скрипт: создаёт пользователя backuper; директории STORAGE_DIR/TRASH_DIR/TEMP_DIR,
#         /var/lib/backuper, /var/log/backuper; кладёт бинарник; ставит .env (600);
#         устанавливает backuper-server.service; backuper-server check-config
sudo systemctl enable --now backuper-server
```

### 24.4 Клиент (Linux)
```
sudo deploy/install.sh client
# директории/пользователь/бинарник/.env(600)/unit; check-config
sudo systemctl enable --now backuper-client
```

### 24.5 Windows (клиент/сервер)
```
1. Скопировать бинарник + .env + сертификаты в C:\Backuper\
2. backuper-*.exe check-config
3. Зарегистрировать службу (sc create / прилагаемый install.ps1), задать автозапуск и recovery
4. Запустить службу
```

### 24.6 Первичный ручной перенос ~1 млн файлов / 4 ТБ
1. Остановить запись в `backup_dir` по возможности (или принять, что часть файлов изменится — это штатно).
2. Перенести данные физически (диск/копирование) в `STORAGE_DIR` на сервере, **сохранив структуру путей 1 в 1**.
3. Установить и запустить клиент и сервер.
4. Сервер при первом запуске проиндексирует `STORAGE_DIR`, запросит список клиента и догонит расхождения (новые/изменённые → скачать, лишние → корзина).
5. Дождаться письма-отчёта со статусом OK/PARTIAL.

### 24.7 Проверка после деплоя
```
backuper-server check-config && backuper-client check-config
backuper-server dry-run        # увидеть план без передачи
backuper-server status
```
